.for each: .forEach() is used to run the same code on each element in an array, but does not change the array and returns undefined. In the example below, we would use .forEach() to iterate through an array of foods and log that we would like to eat each one.

let food = ['mango','rice','pepper','pear']; food.forEach(function(foodItem){ console.log('I want to eat '+foodItem); });

.map(): .map() executes the same code on each element in the array and returns a new array with the updated elements. what is the difference between localStorage ,sessionstorage session and cookies?

Local storage:

Local storage stores large amounts of data on the client computer in key-value format and has no expiration date. The data is never transmitted to the server and is accessible via JavaScript and HTML5.
  • Capacity: 10 MB
  • Accessible from: From any window
  • Expiration: Never
  • Repository Location: Browser only
  • Sent with requirements: No
  • Blockable by users: Yes
  • User Editable: Yes

    • Session storage:

    Session storage stores a large amount of data on the client's computer for the current session only, with the data expiring when the tab is closed. The data is never transferred to the server and is accessible on the client side from the same tab.
  • Capacity: 5 MB
  • Available from: Same card
  • Expiration: When the card is closed
  • Repository Location: Browser only
  • Sent with requirements: No
  • Blockable by users: Yes
  • User Editable: Yes

    • Cookies:

    Cookies store a small amount of data that must be sent back to the server with subsequent requests, and their expiration can be set from the server or the client. They are primarily used for server-side reading.
  • Capacity: 4KB
  • Available from: From any window
  • Expiration: Manually set
  • Repository Location: Browser and Server
  • Sent with requirements: Yes
  • Blockable by users: Yes
  • User Editable: Yes

    • What is Event Loop?

    The event loop is the secret to asynchronous JavaScript programming. JS performs all operations on a single thread, but gives us the illusion of multithreading by using several smart data structures. Let's take a look at what's happening on the back-end. The call stack is responsible for keeping track of all operations in the queue to be performed. Whenever a function completes, it is removed from the stack. The event queue is responsible for sending new functions to the stack for processing. It follows the queue data structure to maintain the correct order in which all operations should be submitted for execution.

    An example of this is the setTimeout method. When a setTimeout operation is processed on the stack, it is sent to the corresponding API, which waits until the specified time to send the operation back for processing.
